# 충돌 정책

## 기본 규칙: union merge ("무조건 둘 다 채택")

`.gitattributes`의 `*.md merge=union` 덕분에, 같은 markdown 파일을 여러 참여자가 동시에 고쳐도 git이 **양쪽 변경을 모두 남긴다**. 충돌 마커(`<<<<<<<`)가 생기지 않는다.

이 정책은 "메모/문서는 지우기보다 남기는 게 안전하다"는 가정에 기반한다.

## 한계 (반드시 이해할 것)

union은 만능이 아니다. 다음은 보장하지 않는다.

1. **중복 제거 안 함** — 두 사람이 각자 같은 문단을 추가하면 문서에 **두 번** 들어간다.
2. **같은 줄 동시 수정** — 한 줄을 양쪽이 다르게 고치면 두 버전이 **둘 다** 남는다(하나로 합쳐지지 않음).
3. **순서 의존 내용** — 논리적으로 순서가 중요한 목록/절차는 병합 후 순서가 뒤섞일 수 있다.
4. **markdown 외 파일** — `*.md`가 아닌 파일에는 union이 적용되지 않는다.

## non-md · 삭제 충돌 폴백 (sync-down)

union이 커버하지 않는 경우는 sync-down 시퀀스가 결정적으로 해소한다.

- **일반 non-md 파일**: `merge -X theirs` → main(원격) 쪽을 채택.
- **수정 vs 삭제**: 정책은 **"수정이 삭제를 이긴다"**.
  - `UD`(우리가 수정, 상대가 삭제) → 우리 것 유지.
  - `DU`(우리가 삭제, 상대가 수정) → 상대 것 복원.
- 그래도 해소 불가하면 `merge --abort` 후 sync-down만 연기(편집은 계속 가능), 상태바에 오류 표시.

## 실무 권고

- 큰 구조 변경(대량 삭제/재배치)은 다른 참여자가 동시에 건드리지 않는 시점에.
- 확정하고 싶으면 **[저장]**으로 main에 스냅샷을 남긴다 — 이후 문제 생겨도 `reflog`/`main` 히스토리로 복구 가능.

## union 드라이버 시드 보장 (모든 연결 경로)

`*.md merge=union` 을 활성화하는 `.gitattributes` 는 **연결 방식과 무관하게** 항상 워킹트리에 시드된다(빈 원격 부트스트랩·기존 wip 재개·기존 원격 adopt 모두). 파일이 없으면 `writeIfAbsent` 로 생성 후 다음 커밋에 흡수되어 main 에 반영된다.

- **기존 파일 존중**: 원격/로컬에 이미 `.gitattributes` 가 있으면 내용을 **변조하지 않는다**(공유 설정 파일을 각 기기가 제각기 덮어쓰는 위험 회피).
- **한계**: 따라서 `merge=union` 이 **없는** `.gitattributes` 를 가진 기존 원격은 union 이 적용되지 않아 `.md` 동시 편집이 `-X theirs`(원격 승)로 떨어질 수 있다. 이 경우 연결 시 로그 경고를 남기며, 해소하려면 원격의 `.gitattributes` 에 `*.md merge=union` 을 직접 추가해야 한다.

## 타이핑 중 원격 변경 (sync-down 연기)

sync-down 의 working-tree 병합(`mergeDown`)은 **유휴(idle) 상태에서만** 실행된다. 느린 fetch 동안 타이핑이 재개되면(마지막 키입력 후 `IDLE_MERGE_MS` 미경과) merge 를 다음 사이클로 **연기**한다 — 그렇지 않으면 병합이 디스크를 다시 써 Obsidian 이 에디터를 리로드하면서 미저장 입력이 소실되기 때문이다. 연기 직전 한 번 더 `flushCommit` 으로 진행 중 편집을 wip 에 흡수해, 다음 사이클 union 병합이 로컬을 보존하도록 한다.

## 온보딩 시 병합 (첫 연결)

이미 파일이 있는 vault를 이미 내용이 있는 원격(main)에 처음 연결하면:

- **파일 단위로는 손실이 없다** — 로컬 전용 파일, 원격 전용 파일 모두 보존된다.
- 다만 **양쪽에 같은 이름이 있고 내용이 다르면 그 파일은 로컬 버전이 채택**된다(첫 adoption 한정). 이후 편집부터는 union 정책이 정상 적용된다.
- 그래서 MVP 권장은 **빈 테스트 vault로 시작**하는 것. 기존 vault를 여러 기기에서 동시에 처음 연결하는 시나리오는 이 한계를 감안한다.
